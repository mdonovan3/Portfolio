Analytics Engineering Pipeline (dbt / Python) — In Progress

A modern data engineering layer built on top of the same PostgreSQL database and source data that powers the operational wine management system. Demonstrates how the same business logic already running in R — POS sales aggregation, GL-based wine COGS — would be structured as a typed, tested, lineage-tracked analytics platform using current DE tooling.

Source data is anonymized and randomized 2024 data from two restaurant locations. The Python ingestion layer replaces the R ETL scripts with an explicit raw landing schema and Parquet intermediates. dbt handles all transformations downstream.

Stack: Python, dbt, SQL, PostgreSQL, Parquet, Node.js, Terraform, AWS EC2, AWS S3, AWS RDS

Pipeline Architecture

Data Sources

  • Aloha POS — DBF files: GNDSALE, GNDITEM, CAT, ITM, EMP
  • Restaurant365 — JSON files: transactions + GL details

Ingestion Layer — Python

  • ingest_aloha.py — DBF → Parquet + raw.aloha_* tables
  • ingest_r365.py — JSON → Parquet + raw.r365_transactions_raw
  • r365_fetch.py — Live OData API → raw schema (daily incremental)

Writes Parquet to data/processed/ and loads to raw.* in PostgreSQL. Idempotent — safe to re-run over the same files.

PostgreSQL — raw schema

Tables: raw.aloha_gndsale, raw.aloha_gnditem, raw.aloha_cat, raw.aloha_itm, raw.aloha_emp, raw.r365_transactions_raw, raw.entities. Data as-ingested. No joins, no business logic. Isolated from production tables.

dbt Staging — views (dbt_staging.*)

  • stg_aloha_sales — TYPE=1 rows, category + employee names resolved
  • stg_aloha_items — Item-level detail, menu item names
  • stg_r365_transactions — Budget filtered, GL parent account derived

Views — no storage cost. Resolve FK lookups, cast types, filter to 2024, exclude bad rows.

dbt Marts — materialized tables (dbt_marts.*)

  • fct_daily_sales — Daily net sales by restaurant + category, rolling 7d / 28d averages
  • fct_r365_wine_cogs — Monthly wine COGS + cost% via 3-part GL formula

dbt tests run after every build — not_null on key columns, warn on known source gaps.

Serving — Node.js Express API

  • GET /api/kpis — KPI snapshot per restaurant
  • GET /api/sales/daily — Daily sales, filterable
  • GET /api/costs/wine — Monthly COGS + cost%

dbt Patterns Used

  • {{ source() }} vs {{ ref() }}: source() references external tables (raw.*) that dbt reads but didn't create. ref() references other dbt models — dbt resolves the dependency graph and builds in the correct order automatically.
  • Schema overrides: dbt_project.yml sets +schema: dbt_staging for staging models and +schema: dbt_marts for marts, regardless of the target's default schema.
  • Source documentation: _sources.yml declares every external table dbt reads from, including column descriptions. Feeds into dbt docs generate — a full data dictionary with lineage graph from a single command.
  • Column-level tests: _marts.yml defines not_null tests on every key column. category_name is set to severity: warn rather than error — known source gap, surfaced visibly rather than silently dropped.

Key Design Decisions

  • Staged raw schema: Data lands in raw.* exactly as ingested — no joins, no business logic. dbt staging models transform from there. If R365 changes a field name, only the staging model changes.
  • Staging as views, marts as tables: Staging models are views (zero storage cost, always current). Mart models are materialized tables — pre-aggregated and fast for the API to query. One dbt run rebuilds the entire chain in under 10 seconds.
  • Critical Aloha composite key: All foreign keys in raw.aloha_* only resolve within the same grind_date. Every join in stg_aloha_* uses entity_id + grind_date as a composite key.
  • Verified COGS formula: Wine cost% = (AP Invoice purchases − AP Credit Memos + inventory asset adjustment) / wine sales — three GL accounts, cross-checked against manual R365 reports before encoding in SQL.
  • Idempotent ingestion: Aloha: DELETE-then-INSERT per entity+date. R365: ON CONFLICT DO NOTHING on (transaction_id, gl_account_number).
  • Portfolio isolation: All pipeline writes go to new schemas (raw.*, dbt_staging.*, dbt_marts.*). No production tables are touched.

Wine Sales ETL — Aloha POS Import

End-to-end ingestion pipeline that extracts nightly Aloha POS DBF exports from each restaurant, resolves location identity from Aloha.ini, and loads wine sales into per-location PostgreSQL databases on AWS RDS. Scheduled via Windows Task Scheduler, written in R.

The source data is genuinely messy — binary DBF format, latin-1 encoding, inconsistent PLU mappings, missing days, and location identity buried in an .ini file rather than the data itself. Handling these reliably and idempotently is the core engineering challenge.

Code: Aloha DBF import gist (functions.R + processDBFs.R)

Pipeline Steps

  • 1. Task Scheduler: wineimport.bat runs nightly on each restaurant's POS server.
  • 2. Folder Validation: folderIsValid() checks for Aloha.ini, GNDLINE.dbf presence, and minimum file size before processing.
  • 3. Location Lookup: UNITNUMBER from Aloha.ini resolves to entity_id and wine_db_name via the entities table.
  • 4. Idempotent Load: importExistsInDB() skips already-loaded dates. Production writes wrapped in dbWithTransaction.

Stats: 3 locations polled nightly, ~1,350 avg records/night, 14 date folders checked, idempotent by date + entity.

Wine Inventory Analysis — How It Works

Period-over-period inventory dashboard. Tracks cost of sales (COS = beginning inventory + purchases − ending inventory), variance (counted vs. theoretical on-hand), purchases, and glass vs. bottle revenue. Built in R Shiny, backed by PostgreSQL on AWS RDS.

  • POS Export: Aloha DBF files read nightly by R ETL client on each restaurant's local machine.
  • PostgreSQL (RDS): Sales, inventory counts, purchases, and theoreticals stored in normalized schema on AWS RDS.
  • Stored Function: get_inventory_activity() and get_theoreticals() aggregate movement by SKU across the period.
  • Shiny Dashboard: R Shiny app queries RDS on date selection, renders value boxes, DT tables, and trend plots.
